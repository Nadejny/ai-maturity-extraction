## 1. OllamaNativeProvider (providers.py)

**Problem:** Ollama's OpenAI-compatible endpoint (`/v1/chat/completions`)
ignores the `think` parameter for Qwen3 models. The model generates
200+ reasoning tokens per document instead of ~10, making inference
10-25x slower.

**Solution:** Added `OllamaNativeProvider` class that uses the native
`/api/chat` endpoint which respects `think: false`.

Key design decisions:
- Strips `/v1` suffix from `base_url` if present (both formats work)
- Sends `stream: false`, generation params in `options` block
  (`num_predict` instead of `max_tokens`)
- Sends `format: json` when `json_mode` is enabled
- Maps `prompt_eval_count`/`eval_count` to standard
  `prompt_tokens`/`completion_tokens`
- Wired in `build_provider()` as `"ollama"` / `"ollama_native"`

## 2. Evidence spans alias resolution (schema.py)

**Problem:** LLM models return evidence_spans with different field names:
- Qwen: `{"text": "...", "start": 0, "end": 145}`
- Schema expects: `{"quote": "...", "start_char": 0, "end_char": 145}`

**Solution:** Added module-level alias tuples and `_first_present()` helper:
```
QUOTE_ALIASES = ("quote", "text", "excerpt", "content", "evidence", "span")
START_ALIASES = ("start_char", "start", "begin", "from", "offset")
END_ALIASES   = ("end_char", "end", "finish", "to", "stop")
```
Inside `_normalize_evidence_spans`, alias resolution runs before all
existing logic (int coercion, negative handling, start/end swap).

## 4. Resume support (inference.py, 03_run_inference.py)

**Problem:** Long inference runs can be interrupted. No way to continue.

**Solution:** Added `--skip-existing` CLI flag and `skip_existing` parameter:
- Loads existing `predictions.jsonl` on resume
- Collects processed `doc_id`s into a set
- Skips already-processed rows
- Merges existing + new predictions for CSV and summary
- Model summary includes `resumed_from_existing_rows` count
- Error counts recomputed from combined list (no double-counting)

## 5. Incremental JSONL writing (inference.py)

**Problem (discovered mid-session):** `predictions.jsonl` was written only
after ALL documents for a model completed. If interrupted mid-model,
all results for that model were lost. `--skip-existing` had nothing
to resume from.

**Solution:** Each prediction is now appended to `predictions.jsonl`
immediately after processing (inside the per-row loop). CSV and summary
are still written at end-of-model (non-critical, regenerated on resume).

Fresh runs truncate leftover files from previous interrupted runs.

**Why not incremental CSV too?** Discussed and decided against it.
JSONL is the primary format (`--skip-existing` reads it), CSV is
secondary (for human review in Excel). CSV is regenerated from the
full prediction list at end-of-model — if the process crashes, CSV
is missing but JSONL has everything. On resume, CSV gets regenerated.

**How the bug was discovered:** During the `qwen_800` run (~40 min in),
we realized the output folder was empty. The process was running but
nothing was on disk yet because `predictions.jsonl` was written only
at end-of-model. We had to wait for qwen3 to finish all 769 docs,
then stop the process (Ctrl+F2 in PyCharm, Ctrl+C didn't work) before
the next model started, to apply the fix.

## 7. Gemma4 thinking mode investigation

**Problem:** `gemma4:e4b` was producing bad/empty outputs and was very slow.

**Root cause found:** Gemma4 supports thinking mode, but there is a
confirmed Ollama bug (ollama/ollama#15260) where `think=false` **breaks
structured output** -- the `format: json` constraint is silently ignored.

**Investigation of all 4 models:**

| Model        | Thinking mode? | think:false safe? |
|-------------|---------------|------------------|
| qwen3:8b    | yes           | yes -- works correctly |
| gemma4:e4b  | yes           | NO -- breaks JSON output (ollama#15260) |
| llama3.1:8b | no            | N/A (ignored) |
| mistral:7b  | no            | N/A (ignored) |

**Fix:** Changed `OllamaNativeProvider` to only send `think: false` for
models in `_THINKING_MODEL_PREFIXES = ("qwen3", "qwq", "deepseek-r1")`.
Gemma4 no longer receives this parameter.

**Trade-off:** Gemma4 defaults to thinking mode enabled (extra latency),
but JSON output is valid. When Ollama fixes #15260, add `"gemma4"` to
the prefix list.

## 9. Prompt rewrite (prompting.py)

Complete rewrite of `build_extraction_prompt()`. Changes:

| Aspect | Old prompt | New prompt |
|--------|-----------|------------|
| System prompt | Generic "extract and return JSON" | "precise information-extraction system", explicit "no markdown, no commentary" |
| Items format | Not specified | Explicit: "FLAT LIST OF SHORT STRINGS", CORRECT/WRONG examples |
| Items limit | Unlimited | Max 5 per field, 2-6 words each |
| Maturity guidance | Just rubric | Conservative thresholds: "Level 3 requires MULTIPLE functions", "when in doubt choose LOWER" |
| Confidence | "must be in [0,1]" | 4-band calibration scale, "do NOT default to 0.95" |
| deployment_scope | Not specified | "describe ACTUAL scope, do NOT copy rubric" |
| evidence_spans | "should contain quotes when available" | "MUST include at least one for every present field" |
| Maturity rationale | Not explained | "1-2 sentences referencing concrete facts, do NOT copy rubric" |
| Few-shot example | None | Complete example (Google Grammar, maturity=2) |
| Structure | Flat text | Sectioned with `=== HEADERS ===` for clarity |

Prompt size grew from ~1KB to ~4KB (excluding article text).

## 10. Mistral diagnosis

- 5/10 docs: partial fields + invented field names
- 2/10 docs: completely off-task (paper metadata, fake API calls)

**Decision:** Keep mistral in the full run. Valuable for the diploma:

## 11. Prompt "overfitting" discussion

Raised a concern: we analyzed qwen3's 769-doc outputs, identified
problems, rewrote the prompt, then tested on 10 docs. Could the prompt
be overfitted to qwen3?

**Conclusion: No.** The prompt changes are generic (format rules,
calibration scales, few-shot examples), not qwen3-specific. All 4
models got the same prompt in the 4x10 test. The improvements
(calibrated confidence, correct items format, no rubric copying)
were observed across all models, not just qwen3. Qwen3 simply
follows instructions better than the others at 8B scale.

The `qwen_800` run is invalidated not because of overfitting but
because it used a different prompt than the other models -- comparing
it with llama/gemma/mistral on the new prompt would be unfair.
