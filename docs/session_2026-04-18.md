## 9. Error tag summary (all models)

| Error tag | qwen3 | llama3.1 | gemma4 | mistral |
|-----------|-------|----------|--------|---------|
| parse_error | 11 | 3 | 13 | 3 |
| evidence_spans_empty_despite_signals | 53 | 73 | 144 | 77 |
| evidence_spans_alias_used | 4 | 86 | 7 | 76 |
| wrong_field_names_detected | 20 | 2 | 43 | 11 |
| maturity_rationale_empty | 0 | 1 | 0 | 3 |

**Parse errors breakdown (qwen3, 11 docs):**
All 11 cases are repetition loops -- the model generates an infinitely
nested JSON structure like `{"ai": {"ai": {"ai": ...}}}` until hitting
the token limit. This is a known failure mode of small LLMs under
constrained JSON generation. The pipeline handles it gracefully
(fallback to empty payload, tagged as `parse_error`).

**Alias usage:** Llama3.1 and Mistral frequently use non-canonical
field names (`"text"` instead of `"quote"`, `"start"` instead of
`"start_char"`). The alias resolution in schema.py normalizes these
transparently. Without it, these would have been data loss.
